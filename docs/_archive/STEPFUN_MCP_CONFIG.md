# 阶跃桌面助手 MCP 配置

## 两种配置方式

阶跃桌面助手支持 **两种模式** 连接 MCP 服务器：

### 方式 1：stdio 模式（npx）

```json
{
  "mcpServers": {
    "mcp-test": {
      "command": "npx",
      "args": ["-y", "<项目路径>"]
    }
  }
}
```

### 方式 2：HTTP URL 模式

```json
{
  "mcpServers": {
    "scrcpy-http": {
      "url": "http://127.0.0.1:3359/mcp"
    }
  }
}
```

## 两种模式对比

| 特性 | npx + stdio | HTTP URL |
|-----|-------------|----------|
| **生命周期** | 每次调用启动新进程 | 连接持续运行的服务 |
| **状态** | 无状态 | 有状态 |
| **延迟** | 高（启动进程开销） | 低（直接连接） |
| **设备连接** | 每次重新连接 | 保持连接 |
| **视频流** | ❌ 不支持 | ✅ 支持 |
| **适用场景** | 轻量工具、测试 | 长连接服务、实时控制 |

## 工作原理

### stdio 模式

```
用户请求 → 阶跃启动 MCP 进程 → 调用工具 → 获取结果 → 关闭进程
              ↑                                       ↑
           启动开销                                进程退出
```

每次调用都是独立进程，适合无状态操作。

### HTTP URL 模式

```
用户请求 → 阶跃连接 HTTP 服务 → 调用工具 → 获取结果
                                      ↑
                              服务持续运行，保持状态
```

连接持续运行的服务，适合有状态操作。

## 适用场景

| 场景 | 推荐模式 | 原因 |
|-----|---------|------|
| 测试工具 | stdio | 轻量、独立、无状态 |
| 一次性操作 | stdio | 截图、点击、输入 |
| 长连接服务 | HTTP URL | 设备控制、视频流 |
| 实时预览 | HTTP URL | 需要持续视频流 |
| 持续监控 | HTTP URL | 需要保持连接状态 |

## 对于 scrcpy-py-ddlx 的建议

scrcpy-py-ddlx 是**长连接架构**，推荐使用 **HTTP URL 模式**：

```json
{
  "mcpServers": {
    "scrcpy": {
      "url": "http://127.0.0.1:3359/mcp"
    }
  }
}
```

优点：
- ✅ 设备连接保持
- ✅ 视频流可用
- ✅ 低延迟
- ✅ 无需代理层

## 安全性思考

### 当前安全架构

```
                    ┌─────────────────────────────────────┐
                    │     scrcpy_http_mcp_server.py       │
                    │                                     │
外部访问 ──────────▶│  HTTP API (8888)    MCP HTTP (?端口) │
                    │                                     │
                    └─────────────────────────────────────┘
                              │
                    ┌─────────┴─────────┐
                    ▼                   ▼
              ┌──────────┐        ┌──────────┐
              │ 手机连接  │        │ AI/GUI   │
              │ HMAC认证  │        │ 无认证   │
              └──────────┘        └──────────┘
```

### 安全链路对比

| 链路 | 认证 | 加密 | 风险等级 |
|-----|------|------|---------|
| 手机 → PC (网络模式) | ✅ HMAC-SHA256 | ✅ 可选 | 中（局域网） |
| MCP → 阶跃 (stdio) | ❌ 无 | ❌ 明文 | 低（本机） |
| HTTP API | ❌ 无 | ❌ 明文 | 中（局域网） |

### 潜在风险场景

| 场景 | 描述 | 严重程度 |
|-----|------|---------|
| 恶意程序扫描 | 本地恶意程序发现端口，调用 API | 中 |
| 局域网嗅探 | HTTP 明文传输可被嗅探 | 中 |
| 恶意 AI 指令 | 用户运行恶意脚本，调用 MCP | 需评估 |
| 命令注入 | 通过 tap/input_text 执行危险操作 | 需评估 |

### 安全设计选项

| 措施 | 说明 | 实现成本 |
|-----|------|---------|
| **本地回环限制** | 只绑定 127.0.0.1，不暴露到局域网 | 低 |
| **MCP Token 认证** | 请求时验证 Bearer Token | 中 |
| **敏感操作确认** | 高危操作需要用户确认 | 中 |
| **操作审计日志** | 记录所有操作便于追溯 | 低 |
| **请求来源验证** | 检查 Origin/Referer | 低 |

### 推荐安全策略

1. **网络层**：HTTP API 只绑定 127.0.0.1
2. **认证层**：MCP 调用需要 Token 验证
3. **操作层**：敏感操作记录日志
4. **审计层**：定期检查操作记录

## Unix Socket 方案（理论可行，但不跨平台）

### 架构

```
┌─────────────────────────────────────────────────────────────┐
│                        PC 端                                │
│                                                             │
│  后台服务（启动一次）                                         │
│  ┌─────────────────────────────────────────────────────┐   │
│  │ scrcpy_http_mcp_server.py                           │   │
│  │   ├── 保持设备连接                                   │   │
│  │   ├── 视频流                                         │   │
│  │   └── 监听 Unix Socket /tmp/scrcpy.sock             │   │
│  └─────────────────────────────────────────────────────┘   │
│                          ▲                                  │
│                          │ Unix Socket (权限 600)           │
│                          │                                  │
│  ┌─────────────────────────────────────────────────────┐   │
│  │ MCP stdio 代理                                       │   │
│  │   ├── 阶跃通过 stdio 调用                            │   │
│  │   ├── 转发到 Unix Socket                            │   │
│  │   └── 进程可随时退出                                 │   │
│  └─────────────────────────────────────────────────────┘   │
│                          ▲                                  │
│                          │ stdio                            │
│                          │                                  │
│                      ┌───────┐                              │
│                      │ 阶跃  │                              │
│                      └───────┘                              │
└─────────────────────────────────────────────────────────────┘
```

### 安全性

| 特性 | 说明 |
|-----|------|
| **无网络端口** | 不暴露 TCP/UDP 端口 |
| **文件权限控制** | chmod 600，只有 owner 可访问 |
| **用户级隔离** | 其他用户无法访问 |
| **进程级隔离** | stdio 只有调用者能访问 |

### 跨平台问题

| 平台 | Unix Socket 支持 | 说明 |
|-----|-----------------|------|
| **Linux** | ✅ 原生支持 | /tmp/scrcpy.sock |
| **macOS** | ✅ 原生支持 | /tmp/scrcpy.sock |
| **Windows** | ❌ 不支持 | 需要使用 Named Pipe 或 TCP |

### Windows 替代方案

| 方案 | 说明 | 安全性 |
|-----|------|--------|
| **Named Pipe** | Windows 原生 IPC | ✅ 高（ACL 控制） |
| **TCP + 127.0.0.1** | 本地回环 | ⚠️ 中（任何进程可访问） |
| **TCP + Token** | 本地 + 认证 | ✅ 较高 |

### 结论

Unix Socket 方案在 Linux/macOS 上可行且安全，但 **Windows 不原生支持**。

跨平台方案需要：
1. 平台检测 + 条件编译
2. Windows 使用 Named Pipe
3. 或接受 TCP + Token 方案

## Headers 测试（已验证 ✅）

**结论：阶跃桌面助手 v0.2.13 支持 `headers` 字段**

### 测试结果 (2026-03-08)

阶跃配置：
```json
{
  "mcpServers": {
    "headers-test": {
      "url": "http://127.0.0.1:3363/mcp",
      "headers": {
        "Authorization": "Bearer test-token-123",
        "X-Custom-Header": "custom-value"
      }
    }
  }
}
```

服务器收到的 headers：
```
authorization: Bearer test-token-12...
x-custom-header: custom-value
mcp-protocol-version: 2024-11-05    ← 阶跃自动添加
user-agent: ...stepfun-desktop/0.2.13...
```

### 安全认证可行

由于阶跃支持 `headers` 字段，Token 认证方案可行：

```json
{
  "mcpServers": {
    "scrcpy": {
      "url": "http://127.0.0.1:3359/mcp",
      "headers": {
        "Authorization": "Bearer <your-secret-token>"
      }
    }
  }
}
```

### 测试工具

| 工具 | 说明 |
|-----|------|
| `check_headers` | 查看服务器收到的所有 headers |
| `verify_auth` | 验证 Authorization header 是否匹配预期 Token |
