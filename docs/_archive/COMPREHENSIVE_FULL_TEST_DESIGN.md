## 一、之前测试的不足分析

### 1.1 测试覆盖不完整

| 遗漏项 | 说明 |
|--------|------|
| **真实 SSE 流** | 之前通过工具调用模拟，未测试真实 `text/event-stream` 连接 |
| **Progress Token** | 未验证 progressToken 的正确传递和跟踪 |
| **批量请求** | 未测试 JSON-RPC 批量请求（数组形式） |
| **错误恢复边界** | 未测试客户端断线重连、网络错误处理 |
| **请求 ID 追踪** | 未系统验证 JSON-RPC id 的正确对应 |
| **协议协商** | 未完整测试 initialize 的 capabilities 交换 |
| **通知顺序** | 未验证通知的到达顺序和时间戳 |

### 1.2 测试日志不完整

之前测试的问题：
- 日志分散在多个文件（txt 记录、md 文档）
- 缺少完整的 JSON-RPC 请求/响应日志
- 无法回放完整的测试过程

### 1.3 测试设计不够严谨

| 问题 | 改进方向 |
|------|----------|
| 单一场景测试 | 增加边界值、异常值、组合场景 |
| 一次性测试 | 增加重复性测试验证稳定性 |
| 无对比参照 | 增加 HTTP vs stdio 模式对比 |
| 缺少断言 | 增加明确的通过/失败判断标准 |

## 三、全量测试设计

### 3.1 测试服务器设计

创建统一的全量测试服务器，具备以下能力：

1. **完整日志记录**: 每个请求/响应记录到 JSONL 文件
2. **请求 ID 追踪**: 验证 JSON-RPC id 正确对应
3. **时间戳记录**: 记录请求到达和响应发送的精确时间
4. **批量请求支持**: 支持 JSON-RPC 批量请求测试
5. **通知发送**: 支持发送各种通知并记录
6. **状态追踪**: 记录会话状态和能力协商结果

### 3.2 测试用例设计原则

1. **边界值测试**: 最小值、最大值、空值、超长值
2. **异常值测试**: 无效类型、格式错误、越界值
3. **并发测试**: 同时发送多个请求
4. **时序测试**: 验证事件顺序和时间间隔
5. **重复性测试**: 多次执行验证稳定性
6. **对比测试**: HTTP vs stdio 模式对比

### 3.3 测试分类

#### A. 核心能力测试（P0）- 必须全部通过

| 编号 | 测试项 | 子测试 | 验证点 |
|------|--------|--------|--------|
| A1 | 基础连通性 | ping | pong 返回、时间戳 |
| A2 | 工具列表 | tools/list | 返回格式、工具数量、schema 完整性 |
| A3 | 工具调用 | tools/call | 参数传递、返回格式、错误处理 |
| A4 | 协议初始化 | initialize | 版本协商、能力交换、信息完整性 |

#### B. 重要能力测试（P1）- 大部分应通过

| 编号 | 测试项 | 子测试 | 验证点 |
|------|--------|--------|--------|
| B1 | 复杂参数 | 嵌套对象、数组、枚举 | 参数解析正确性 |
| B2 | 大数据量 | 100/1000/10000 项 | 内存、性能、超时 |
| B3 | 长时间操作 | 10s/30s/60s | 超时处理、进度通知 |
| B4 | 并发请求 | 3/5/10 并发 | 结果正确性、顺序性 |
| B5 | Unicode | 中文、日语、阿拉伯语、Emoji | 编码正确性 |
| B6 | 错误处理 | 各类错误码 | 错误识别、恢复能力 |
| B7 | 资源 API | list/read | 返回格式、内容正确性 |
| B8 | 提示 API | list/get | 返回格式、内容正确性 |
| B9 | 进度通知 | notifications/progress | 通知接收、顺序、内容 |
| B10 | 取消通知 | notifications/cancelled | 取消生效、状态 |
| B11 | 采样请求 | sampling/createMessage | 回调响应、结果内容 |

#### C. 高级能力测试（P2）- 部分可能不支持

| 编号 | 测试项 | 子测试 | 验证点 |
|------|--------|--------|--------|
| C1 | SSE 流式 | text/event-stream | 连接建立、事件流、关闭 |
| C2 | 批量请求 | JSON-RPC 数组 | 请求处理、响应匹配 |
| C3 | 分页 | cursor/limit | 游标正确性、边界处理 |
| C4 | 资源订阅 | subscribe/updated | 订阅生效、通知接收 |
| C5 | 自动补全 | completion/complete | 建议正确性、匹配度 |

#### D. 边界条件测试（P3）- 稳定性验证

| 编号 | 测试项 | 子测试 | 验证点 |
|------|--------|--------|--------|
| D1 | 空参数 | {} | 默认值处理 |
| D2 | 超长参数 | 10KB/100KB 字符串 | 截断、拒绝、超时 |
| D3 | 特殊字符 | 控制字符、引号、换行 | 转义正确性 |
| D4 | 重复请求 | 相同请求 10 次 | 结果一致性 |
| D5 | 快速连续 | 100ms 间隔连续请求 | 连接稳定性 |

## 四、测试执行流程

### 4.1 准备阶段

1. 启动全量测试服务器（带完整日志）
2. 在阶跃桌面助手配置 MCP 服务器
3. 清空之前的日志文件

### 4.2 执行阶段

按 A → B → C → D 顺序执行：
1. 每个测试用例发送请求
2. 记录请求/响应到日志文件
3. 验证响应是否符合预期
4. 记录测试结果

### 4.3 分析阶段

1. 解析日志文件
2. 统计通过率
3. 分析失败原因
4. 生成测试报告

## 五、日志格式设计

### 5.1 请求日志

```json
{
  "timestamp": "2026-03-09T15:00:00.000000",
  "type": "request",
  "request_id": 1,
  "direction": "inbound",
  "jsonrpc": "2.0",
  "method": "tools/call",
  "id": "req-001",
  "params": {...},
  "http": {
    "method": "POST",
    "path": "/mcp",
    "headers": {...},
    "content_type": "application/json"
  }
}
```

### 5.2 响应日志

```json
{
  "timestamp": "2026-03-09T15:00:00.500000",
  "type": "response",
  "request_id": 1,
  "direction": "outbound",
  "jsonrpc": "2.0",
  "id": "req-001",
  "result": {...},
  "elapsed_ms": 500
}
```

### 5.3 通知日志

```json
{
  "timestamp": "2026-03-09T15:00:01.000000",
  "type": "notification",
  "direction": "outbound",
  "jsonrpc": "2.0",
  "method": "notifications/progress",
  "params": {
    "progressToken": "xxx",
    "progress": 0.5,
    "total": 1.0
  }
}
```
